# QDBT 비교 단위 트리 — 전체 설계서

작성일: 2026-06-25
대상: 비교 단위(group by) 트리, 그룹별 깊이, 재클러스터링, LLM 제안, 업체 비교
상태: 설계 확정안. 1단계(전체 레벨 group by)는 v0.8.9에 구현됨.

---

## 0. 핵심 철학 — 파워피벗 모델

**데이터(잎)는 가장 상세하게 보존. 비교 단위는 "보기 방식"으로 분리.**

- 추출은 가장 깊은 잎까지 저장 (3단계든 4단계든)
- 비교 단위 = 트리에서 "어느 깊이에서 멈추느냐"
- 멈춘 지점이 바뀌어도 원본 데이터는 불변
- 따라서 비교 단위는 언제든 자유롭게 변경 가능

이것이 전체 설계를 관통하는 원칙이다.

---

## 1. 데이터 구조 (DB)

### 이미 있는 것 — 트리는 path로 표현됨
submission_items 테이블:
- `path`     — 전체 경로 "재료비 > 기구부 > 차폐 > 납Plate" = 트리 그 자체
- `depth`    — 깊이
- `line_no`  — 원본 표기 (1.1.3.1)
- `sort_order` — 원본 순서
- `name_normalized`, `amount`, `unit`, `unit_price` 등

→ **트리는 별도 저장하지 않는다.** 각 잎이 자기 전체 경로를 가지므로,
   path들을 모아 트리를 재구성한다.

### 추가할 것 — 비교 단위(펼침 상태) 저장
현재 v0.8.9: submissions.compare_level (단일 숫자, 전체 한 레벨).
→ 그룹별 다른 깊이를 담으려면 확장 필요.

**방식: 비교 단위 경로 집합을 JSON으로 저장**
submissions.compare_units (TEXT, JSON):
```json
["재료비 > 기구부 > 차폐",
 "재료비 > 기구부 > 프레임",
 "재료비 > 전장/제어부",
 "이윤 및 관리비"]
```
= "이 경로들이 비교 단위다" (트리에서 멈춘 지점들).
- 그룹마다 깊이가 달라도 표현 가능 (기구부는 소분류, 이윤은 대분류)
- compare_level(단일 숫자)는 하위호환으로 유지하거나 폐기
- 잎 데이터는 불변, 이 컬럼만 "보기 설정"으로 추가

### 업체별 독립 — 추출 화면 vs 비교 화면
같은 업체라도 두 화면이 각자 펼침 상태를 가질 수 있음:
- submissions.compare_units      — 추출 검토 시 펼침
- (비교 화면) bid 단위로 별도 저장 또는 세션
→ 추출 시 조정한 깊이와 비교 시 관점이 다를 수 있으므로 분리.

## 2. 비교 단위 트리 UI

### 동작 (일반 트리, +/− 펼침)
- 각 가지 +/− 로 펼침/접기, 가장 하위(잎)까지 계단식 드릴다운
- **"더 펼칠 수 없는 노드" = 비교 단위** (파란 배지):
  · 잎(말단) → 항상 비교 단위
  · 접힌 가지 → 비교 단위
  · 펼친 가지 → 비교 단위 아님 (그 아래가 단위)
- 접힌 가지 금액 = 하위 전체 합산
- "모두 펼치기 / 모두 접기" 버튼

### 그룹별 다른 깊이 (핵심)
한 화면에서 그룹마다 다른 깊이로 멈출 수 있음:
- 재료비 > 기구부 → 소분류(차폐/프레임/셔틀)까지 세밀하게
- 이윤 및 관리비 → 대분류로 접어서 큼직하게
→ 멈춘 지점들의 집합이 곧 compare_units.

목업: qdbt_tree_mockup.jsx (검증됨)

## 3. 재클러스터링 — 비교 단위 기준

### 가능. 구조가 이미 받침.
현재 클러스터링: _build_cluster_input()이 품목 리스트(잎, name_normalized)를
LLM 입력으로 만듦 → 품명 레벨 클러스터링.

**교체 지점: input을 "비교 단위 묶음"으로 바꾼다.**
```
[현재]  잎 품목들 → _build_cluster_input → LLM → 품명 클러스터
[변경]  비교단위 묶음(명칭+합산금액) → 같은 input 함수 → LLM → 그 레벨 클러스터
```

흐름:
```
트리에서 비교 단위 결정 (compare_units)
   ↓
group_items_by_units(): 각 비교 단위의 [대표명칭 + 합산금액 + 멤버] 산출
   ↓ (이미 만든 group_items_by_level의 일반화)
이 묶음을 클러스터링 input으로
   ↓
그 레벨로 클러스터링됨
   ↓
비교 단위 바꾸면 → 다시 묶어 → 재클러스터링
```

→ **데이터(잎) 불변, "어느 묶음을 클러스터링하나"만 교체.**
   레벨 상/하위 어디로든 재클러스터링 자유.

### 저장 시점 — 보기는 자유, 확정은 명시적
- 레벨 바꿔가며 보는 것: 임시 (저장 안 함)
- 사용자가 "이 클러스터링 확정" → catalog_clusters에 저장
- (파워피벗처럼 탐색은 가볍게, 커밋은 의도적으로)

## 4. LLM 비교 단위 제안 (NEW 핵심)

### 목적 — 사용자 노동 절감
업체마다 트리를 일일이 펼치는 부담 → LLM이 "이 업체는 이 깊이가
비교에 적절"이라고 1차 제안 → 사용자는 검토·조정만.

### "A의 소분류 = B의 중분류" 문제를 LLM이 해결
업체마다 분류를 쪼개는 정도가 다름:
  A: 대>중(전장부)>소(센서)       ← 센서가 소분류
  B: 대>중(센서)>소(온도/압력센서)  ← 센서가 중분류
LLM은 "센서"가 의미상 같은 비교 대상임을 알아챔 (글자매칭 불가).
→ 업체별로 "센서에서 만나도록" 레벨을 제안.

### 작동
```
입력: 여러 업체의 트리(path들) + 금액
프롬프트: "업체 간 비교를 위해, 각 업체에서 어느 분류 레벨이
          적절한 비교 단위인지 제안하라. 같은 의미 항목이
          업체마다 다른 레벨일 수 있음을 고려하라."
출력(JSON): 업체별 × 그룹별 권장 비교 단위
   ↓
트리의 "초기 펼침 상태"로 반영
   ↓
사용자가 검토·조정 (+/−)
```

### 원칙 — LLM은 제안, 사람이 결정
- LLM이 자동 확정 ✗ (오판 위험, 비즈니스 판단은 사람 책임)
- 클러스터링 accept/reject와 동일 패턴: 제안 → 검토 → 확정
- 기존 클러스터러에 LLM 호출 구조 있음 → 유사 호출 추가

## 5. 업체 간 비교 화면

### 구성
여러 업체를 나란히, 같은 비교 단위끼리 금액 비교:
```
비교 단위        A업체      B업체      C업체
기구부          500만      450만      480만
전장/제어부      300만      320만        -
이윤및관리비     75만       80만       70만
```

### 업체별 레벨 다름 → 명칭으로 매칭
- 각 업체가 지정/제안된 비교 단위로 묶임 (레벨 다를 수 있음)
- 묶인 명칭+금액으로 클러스터링 → 같은 것끼리 한 행에
- A엔 있고 C엔 없으면 "-" (누락 표시)

### 관점 조정
비교 화면에서도 트리 펼침 조정 가능 → 추출 화면과 다른 관점.

## 6. 전체 흐름 (종합)

```
업체별 추출 (청킹 + 병합복원 → path 채워짐)
   ↓
[추출 검토 화면]
   비교 단위 트리 (+/− 펼침, 그룹별 깊이)
   LLM 비교단위 제안 → 초기 펼침
   사용자 조정 → compare_units 저장
   ↓
[단일 업체 재클러스터링]
   비교 단위 묶음 → 클러스터링 input → 그 레벨로 묶음
   확정 시 catalog_clusters 저장
   ↓
[업체 간 비교 화면]
   여러 업체 나란히, 비교단위로 매칭
   LLM이 업체별 레벨 제안 (A소분류=B중분류 맞춤)
   명칭+금액 클러스터링 → 금액 비교
   관점 조정 가능
```

## 9. ★ 계층 전파 — 네 군데 일관 (추출/비교/클러스터/카탈로그)

### 진짜 목적: 계층 비교·분석·리콜을 미래 입찰에 재사용
비교 단위 트리가 추출 화면에서만 살면 의미 없음.
클러스터·카탈로그까지 계층이 전파되어야:
- 과거 카탈로그를 기준으로 새 입찰 비교 (시계열 리콜)
- LLM이 레벨 제안 시 "과거에 어느 레벨로 쌓았나" 참조
- 카탈로그 레벨 재조정 (예: '기구부'로 쌓다가 '차폐'로 내려보기)

### 현재 DB 상태 (점검 결과)
| 영역 | 계층 받침 | 비고 |
|------|----------|------|
| submission_items | ✅ path, depth, line_no | 견고 |
| catalog_categories | △ parent_id 트리 있음 | 추출 path와 미연결 |
| catalog_items | ❌ category_id 단일 | 평면. 어느 잎인지 모름 |
| catalog_clusters | ❌ 레벨 정보 없음 | 품명 레벨 고정 |
| price_history | ❌ catalog_item_id만 | 레벨별 리콜 불가 |
| submissions | △ compare_level만 | 그룹별 깊이 못 담음 |

→ 뼈대(path, parent_id)는 있으나 연결·전파·재조정이 빠짐.

### 카탈로그도 트리여야 한다 (핵심 결정)
이유: 미래 재사용 + 레벨 재조정 (기구부→차폐).
파워피벗 group by를 카탈로그에도 적용:
- 카탈로그 항목이 잎까지 계층(path) 보존
- 카탈로그 자체도 레벨 재조정 가능
- 잎 보존 → 나중에 다른 레벨로 재구성

### DB 보강 (전파를 위한)
```
submissions
  + compare_units (JSON)      -- 비교 단위 경로 집합 (그룹별 깊이)

catalog_items
  + canonical_path (TEXT)     -- 이 표준품목의 분류 경로
  + depth (INTEGER)           -- 계층 깊이
  + observed_paths (JSON)     -- 입찰마다 나타난 여러 경로 (다중)
       ※ 같은 품목이 A입찰=재료비>센서, B입찰=전장부>센서일 수 있음
         → 단일 고정 대신 관측된 경로들을 누적 (정확한 리콜)

catalog_clusters
  + compare_unit_path (TEXT)  -- 이 클러스터의 비교 단위 경로
  + compare_level (INTEGER)   -- 어느 레벨로 클러스터링했나
       ※ 클러스터가 "어느 트리 단위"인지 기록 → 레벨별 재클러스터 추적

price_history
  + item_path (TEXT)          -- 당시 항목의 분류 경로 스냅샷
  + compare_unit (TEXT)       -- 어느 비교 단위로 기록됐나
       ※ 레벨별 가격 리콜 가능. "기구부 가격 추이" / "차폐 가격 추이" 둘 다.
```

### LLM 레벨 제안이 카탈로그 참조
```
새 입찰 비교 단위 제안 시:
  입력: 새 입찰 트리 + [기존 카탈로그의 계층/레벨]
  LLM 판단: "과거 카탈로그가 '기구부' 레벨로 쌓여있다
            → 이 업체도 '기구부'로 맞춰야 시계열 비교 가능"
  출력: 카탈로그와 정합하는 레벨 제안
→ 카탈로그 계층이 LLM 제안의 기준. 레벨 일관성 유지 → 리콜 가능.
```

### 카탈로그 레벨 재조정
```
현재: 카탈로그가 '기구부' 레벨로 구성됨
미래: '차폐' 레벨로 더 잘게 보고 싶음
→ canonical_path/observed_paths가 잎까지 보존돼 있으면
  '기구부' → '차폐'로 재group by 가능 (파워피벗처럼)
→ price_history.item_path도 잎까지 있으면 과거 가격도 재집계
```

## 핵심 요약
1. 트리는 path로 이미 표현됨. 비교 단위(펼침)는 compare_units JSON 추가.
2. 데이터(잎) 불변, 비교 단위는 보기 설정. 파워피벗 모델.
3. 재클러스터링: 비교 단위 묶음을 클러스터링 input으로 교체. 어느 레벨로든 자유.
4. LLM이 업체별 적절 레벨 제안 (제안→검토→확정). "A소분류=B중분류" 해결.
5. 업체 비교: 레벨 달라도 명칭으로 매칭.
6. ★ 계층은 네 군데(추출/비교/클러스터/카탈로그) 전파되어야 함.
   카탈로그도 트리. 잎까지 보존 → 레벨 재조정·시계열 리콜 가능.
7. ★ LLM 레벨 제안은 기존 카탈로그 계층을 참조 → 레벨 일관성 → 미래 재사용.

계층 전파는 나중에 끼우기 어려움 (데이터가 쌓인 뒤엔 마이그레이션 부담).
→ 트리/클러스터 UI 만들기 전에 DB 스키마부터 계층 전파 가능하게 확장.
   특히 price_history.item_path는 처음부터 잎 경로를 기록해야
   나중에 레벨별 리콜이 가능 (소급 불가).
